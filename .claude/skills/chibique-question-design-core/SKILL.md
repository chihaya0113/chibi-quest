# chibique-question-design-core — 問題設計の共通ルール

このSkillは「どんな問題を作るか」を決める。「どう書くか(表記)」「どう反映するか(生成スクリプト・audit)」は kids-content Skill が正本。両方に従うこと。

## 1. 設計シート(問題を書く前に必ず埋める)

単元ごとの作成・刷新に着手する前に、以下を箇条書きで書き出す(コメントやPR説明に残す)。埋まらない項目がある状態で問題文を書き始めない。

1. 対象学年: 例「小3」
2. 教科・単元: 例「math / division_basic」
3. 測る能力: 例「わり算を『分ける場面』と結びつけて式を立てられるか」
4. 使ってほしい考え方: 例「全体÷1つ分=いくつ分、を場面から選ぶ」
5. 想定難易度: §2の5軸で決める(総合1〜5)
6. 解答形式: numeric_input / multiple_choice / expression_choice のいずれか(新形式はアプリ側の対応が必要なので、追加する場合は先にユーザーへ提案する)
7. 解いた後に身につくこと: 例「かけ算とわり算を逆向きに使い分けられる」
8. 面白さの核: §4のカタログから1つ選ぶ(「なんとなく楽しい」は不可)

## 2. 難易度の設計(5軸)

難易度は1つの数字で決めず、以下の5軸を各1〜3点で先に決め、合計から総合difficultyを出す。

| 軸 | 1点 | 2点 | 3点 |
|---|---|---|---|
| 知識 | 単元の代表知識そのまま | 単元内の応用 | 既習の他単元と組み合わせ |
| 情報量 | 数値・条件が1つ | 2つ | 3つ以上 or 不要な情報の選別が要る |
| 思考ステップ | 1手 | 2手 | 3手以上 or 自分で式を組み立てる |
| 形式の慣れ | 定番の問い方 | 問う向きが逆(答え→条件など) | 初見の形式(並べ替え、誤り探し等) |
| 選択肢の近さ | 明確に区別できる | ありがちな誤解が1つ混ざる | 全選択肢がありがちな誤解 |

- 総合difficulty: 合計5〜6→1、7〜8→2、9〜10→3、11〜12→4、13〜15→5
- 各単元は difficulty 1〜4 を分布させる(目安: 1が2割、2が3割、3が3割、4が2割。5は単元の仕上げ用に数問まで)
- **難易度を上げる禁止手段**: 文章を長くする / ひっかけ・意地悪な言い回し / 未習内容(小3範囲外)を混ぜる / 日本語の曖昧さで迷わせる。難しくするなら上の5軸(特に思考ステップ・形式の慣れ)で上げる。

## 3. 類題・重複防止

- 同じ単元で新しい問題を作るとき、既存のどの問題と比べても**次のうち2つ以上**が異なること。数値の差し替えだけは1問も作らない。
  - 状況(買い物→図書室、など)
  - 問う向き(答えを求める ⇄ 条件を求める)
  - 与える情報(式を与える ⇄ 場面だけ与える)
  - 解答形式(numeric_input ⇄ multiple_choice 等)
  - 認知活動(正解を出す / 誤りを見つける / 理由を選ぶ)
- ほぼ同型の問題(意図的なドリル反復)は同じ `familyId` を付け、**1 familyにつき5問まで**。出題側がfamilyIdで連続出題を避ける前提。
- ただし表現を変えた結果、設計シートの「測る能力」が別物になっていないか確認する。変わっていたら別問題として設計シートから作り直す。

## 4. 面白さの核(funMechanicカタログ)

各問題に以下のいずれか1つを `funMechanic` として割り当てる。「解く過程に小さな予想・発見・判断があること」が面白さの定義。

| 値 | 内容 |
|---|---|
| predict_check | 先に予想させてから確かめる |
| find_mistake | まちがいを見つける・直す |
| best_choice | どれも一応正しい中から最適を選ぶ |
| inference | 少ない手がかりから推理する |
| reorder | 並べ替えて意味・手順を完成させる |
| rule_discovery | 規則・法則を見つける |
| compare_methods | 2つの解き方・考え方を比べる |
| judge_claim | キャラクター2人の主張のどちらが正しいか判断する |
| decide_then_verify | 自分ならどうするか選び、教科知識で確かめる |
| drill | 反復練習(核なし) |

- drill は基礎定着に必要なので禁止しないが、**各単元の4割以下**にする。
- 各単元で drill 以外のメカニクスを**3種類以上**使う。
- 演出・ストーリーは問題文の前置き**2文以内**。それ以上必要な設定は問題に向いていないので作り直す。

## 5. 文章ルール

表記(漢字・文長・励まし文言)は kids-content Skill の表記ルールに従う。このSkillでの追加ルール:

- 正解が2通りに読める文を書かない。書いた後「別の解釈で別の答えが出ないか」を必ず自問する。
- 「それ」「これ」等の指示語は、直前の1語を指す場合以外使わない。
- 問題を解くのに関係ない固有名詞・設定を足さない(登場人物は最大2人)。
- 二重否定・否定形の問い(「正しくないものはどれ」)は各単元1割以下。使う場合は「正しくない」を太字等で強調できないため、文頭に置く。
- 読解を測る問題(国語の読み取り)以外では、文章の読みにくさで正答率を下げない。

## 6. 教科別の面白さの軸

共通品質(§1〜5)を守った上で、教科ごとに次の軸を優先する。

- 国語: 文脈から意味・根拠を見つける(「どの一文からわかる?」)
- 算数: 条件を整理して法則・解き方を発見する(「どう分ければいい?」)
- 理科: 予想→確かめ、因果関係(「こうしたらどうなる?」)
- 社会: 地図・資料・立場の違いを読み解く(「この人の仕事はなぜ必要?」)
- 英語: 意味のある場面で伝わる表現を選ぶ(「この場面でなんて言う?」)

## 7. サッカー・世界観の使い方

- サッカー文脈は**自然に合う場合だけ**使い、各単元の**3割以下**。残りは学校・家庭・買い物・自然・地域・冒険などを使う。
- サッカーの知識(ルール・選手名)が正解を左右する問題は作らない。サッカーを知らない子でも解けること。
- 教科の学習目標がサッカー演出より常に優先。演出を足して§5に反する長さになるなら演出を削る。
- 正解が試合・育成に小さく良い影響を与える仕組み(pt等)への接続は歓迎するが、アプリ側の変更を伴う場合は先にユーザーへ提案する。

## 8. 品質チェックリスト(問題ごと)

生成前のレビューで全問について確認する。1つでもNGなら修正してから audit へ進む。

- [ ] 正解が一意(§5の「別の解釈」チェック済み)
- [ ] 問題文と選択肢だけで解答できる(外部知識・画像への依存なし)
- [ ] 学習内容として正確(算数・理科は数値を再計算した / 英語は文法・自然さを確認した / 国語・社会は事実関係を確認した)
- [ ] 小3で未習の知識を要求していない(範囲外は kids-content の手順どおりユーザー確認)
- [ ] 問題文と解説が矛盾しない。解説は「なぜそうなるか」を含む
- [ ] 誤答選択肢は子供がしそうな誤解の反映(kids-content 4択基準)で、形式的ヒント(長さ・語尾)がない
- [ ] 既存問題と比べ§3の「2つ以上の違い」がある
- [ ] 難しさが5軸から来ている(長文化・ひっかけ由来でない)
- [ ] funMechanic が1つ割り当てられ、解く過程に予想・発見・判断のいずれかがある
- [ ] 不快・不安・差別的な内容、特定の子を傷つけうる設定(病気・貧困・容姿等)がない

大量生成時は、全問チェックに加えて**単元ごとに無作為に5問**を選び、実際に自分で解いてチェックリストを再適用する。

## 9. データ出力仕様

既存スキーマ(src/data/questions/grade3 の実データ)が正本。新規フィールドの追加仕様と全フィールド一覧は [references/metadata-schema.md](references/metadata-schema.md) を参照。要点:

- 既存フィールド(id, subject, grade, unit, difficulty, questionType, prompt, choices, answer, explanation, estimatedSeconds, skillTags, status 等)は形式を変えない。
- このSkillで新たに必須化: `familyId`(算数・国語にも付与)、`learningObjective`、`difficultyAxes`、`funMechanic`、`commonMistake`。
- `hint` は推奨(現状アプリに表示UIがないため、実装提案とセットで)。
- フィールドを追加したら scripts/audit-questions.mjs の検証も同時に更新し、新フィールドの欠落を検出できるようにする。

## 禁止事項

- 設計シート(§1)を書かずに問題文から書き始める
- 数値だけ差し替えた類題を作る
- 難易度調整を文章の長文化・ひっかけで行う
- funMechanic を後付けでそれらしく選ぶ(設計シート時点で決める)
- kids-content の audit・生成スクリプト手順を飛ばす
